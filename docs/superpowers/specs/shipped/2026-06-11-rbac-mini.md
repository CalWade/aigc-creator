# RBAC mini — User.role 字段 + 三层门控

**日期**:2026-06-11
**前置状态**:phase-2.6 起,admin 鉴权走 `ADMIN_HANDLES` env 白名单
**结束状态**:权限源收敛到 DB `User.role`(enum AUTHOR/ADMIN),env 白名单全删

## 背景

phase-2.6 上 admin 工作台时,临时用 `ADMIN_HANDLES=admin` 环境变量做白名单。这套方案有两个问题:

1. **前端无门控** — sidebar / command-menu 对所有人无条件渲染"管理"分组,普通作者点进去看到"加载失败"(后端 403 兜底),不是"无权访问"。体验割裂。
2. **权限源分裂** — User 表无 role 字段,admin 身份由 env 决定。schema 与运行时不对齐,扩展(多角色 / 角色变更审计)受限。

## 决策

| 项              | 旧                                           | 新                                                                       |
| --------------- | -------------------------------------------- | ------------------------------------------------------------------------ |
| 权限源          | `process.env.ADMIN_HANDLES`(逗号分隔 handle) | `User.role` enum(AUTHOR/ADMIN)                                           |
| JWT payload     | `{ sub, handle }`                            | `{ sub, handle, role }`                                                  |
| AdminGuard 判断 | `req.user.handle in env.split(",")`          | `req.user.role === "ADMIN"`                                              |
| 前端门控        | 无                                           | 导航层 + 路由层 + 接口层(深度防御)                                       |
| 老 token 兼容   | n/a                                          | payload 无 role → fail-closed 拒绝;前端 AuthUser.role 缺失 → 视为 AUTHOR |

## 三层门控

1. **导航层** — `apps/web-studio` 的 `sidebar-nav.tsx` / `command-menu.tsx` 通过 `useAuthSnapshot()` 拿 `user.role`,role !== ADMIN 时整个"管理"分组不渲染。普通作者看不到入口。
2. **路由层** — `apps/web-studio/src/app/(creator)/admin/layout.tsx` 客户端守卫:非 ADMIN → `window.location.replace("/me/dashboard")`。手敲地址也防住。守卫期间渲染"正在校验权限..."骨架避免闪屏。
3. **接口层** — 后端 `AdminGuard` 兜底,所有 `/admin/*` 端点走 `JwtAuthGuard + AdminGuard` 双重保护。即使前端两层都被绕过,后端也会 403。

## 数据迁移

- migration `20260611045252_add_user_role`:`CREATE TYPE "UserRole" AS ENUM ('AUTHOR', 'ADMIN'); ALTER TABLE "User" ADD COLUMN "role" "UserRole" NOT NULL DEFAULT 'AUTHOR';`
- fixture 4 个 demo 账户:`admin`=ADMIN,`demo-author`/`tech-author`/`life-author`=AUTHOR
- 部署时:`prisma migrate deploy` 在容器启动前跑;产线如有 admin 用户需 SQL 显式 `UPDATE "User" SET role='ADMIN' WHERE handle IN (...)`(本仓库还没产线)

## 老 token 行为

升级后所有现存 JWT payload 无 role 字段 → `req.user.role === undefined` → AdminGuard fail-closed 拒绝。

**用户体感**:

- 老 admin 必须重新登录才能进 `/studio/admin/*`(再签时新 payload 带 role)
- 普通作者无感(他们本来也访问不了 admin)

## 删除清单

`ADMIN_HANDLES` 共 6 处全清:

- `apps/api/src/config/env.validation.ts` Joi schema
- `apps/api/test/setup-env.ts` jest env 注入
- `apps/api/.env.example` / `apps/api/.env`
- `deploy/.env.example` / `deploy/docker-compose.yml`
- `.github/workflows/ci.yml` 注释

`grep -r "ADMIN_HANDLES"` 应只命中 `docs/superpowers/{specs,plans}/shipped/*` 历史快照(保留)。
